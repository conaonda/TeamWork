# 워크플로우 가이드

## 전체 흐름

```
이슈 생성 → 브랜치 생성 → 개발 → PR 생성 → 코드리뷰 → 머지
```

## 1. 이슈 생성
- 모든 작업은 이슈에서 시작합니다.
- 적절한 템플릿(버그/기능/태스크)을 선택합니다.
- 라벨과 담당자를 지정합니다.

## 2. 브랜치 생성
- `develop` 브랜치에서 새 브랜치를 생성합니다.
- 네이밍 규칙:
  - 기능: `feature/이슈번호-간단한설명` (예: `feature/42-user-login`)
  - 버그: `fix/이슈번호-간단한설명` (예: `fix/15-header-overflow`)
  - 핫픽스: `hotfix/이슈번호-간단한설명`

## 3. 개발
- [커밋 컨벤션](commit-convention.md)을 따릅니다.
- 작은 단위로 자주 커밋합니다.
- 의미 있는 커밋 메시지를 작성합니다.

## 4. PR 생성
- PR 템플릿을 작성합니다.
- 관련 이슈를 연결합니다 (`closes #이슈번호`).
- 리뷰어를 지정합니다.

## 5. 코드리뷰
- [코드리뷰 가이드](code-review-guide.md)를 참고합니다.
- 최소 1명(또는 1 에이전트)의 승인이 필요합니다.
- 리뷰 응답 시간: 에이전트는 즉시 처리, 사람은 24시간 이내.

## 6. 머지
- 리뷰 승인 후 Squash Merge를 사용합니다.
- 머지 후 브랜치를 삭제합니다.

## 릴리스
- `develop` → `main`으로 PR을 생성하여 릴리스합니다.
- 태그를 생성합니다 (예: `v1.0.0`).

## AI 에이전트 팀
AI 에이전트로 구성된 팀의 경우 [에이전트 워크플로우](agent-workflow.md)를 참고하세요.
